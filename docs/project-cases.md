# Reliable Webhook 项目案例

这个项目不要讲成“我写了一个 Webhook 平台”。更准确的讲法是：

> 我做了一个可靠 Webhook 投递服务，用来解决业务系统向外部系统发送事件通知时，网络失败、对方服务异常、进程重启导致通知丢失或无法追踪的问题。

## 案例一：订单创建后通知第三方系统

### 真实问题

电商系统创建订单后，需要通知第三方 CRM、ERP 或商家系统。

如果订单服务直接调用第三方接口，会遇到几个问题：

- 第三方服务短暂不可用，订单系统不能一直阻塞。
- HTTP 请求超时后，不知道对方到底有没有收到。
- 服务重启后，内存里的待通知任务会丢失。
- 出问题后只能查日志，很难知道某个订单通知到底失败了几次。

### 这个项目怎么解决

订单服务只负责把事件写入 `events` 表，并在同一个事务里创建 `deliveries` 投递任务。

后台 worker 异步投递：

```text
pending -> running -> succeeded
pending -> running -> pending
pending -> running -> dead
```

如果第三方返回 `500` 或请求超时，任务会进入下一轮重试。

如果返回 `400` 这类非重试错误，任务会进入 `dead`，避免无意义重试。

### 能体现的能力

- 能把同步调用改造成异步可靠投递。
- 能用数据库事务保证事件和投递任务一起创建。
- 能区分可重试失败和不可重试失败。
- 能通过状态机描述复杂流程。

## 案例二：支付成功事件不能丢

### 真实问题

支付成功后，系统需要通知会员系统、订单系统或营销系统。

支付事件通常不能丢，但下游系统可能会临时失败。

直接发 HTTP 请求的问题是：

- 失败后没有统一重试机制。
- 重试次数和失败原因不可见。
- 多个系统各自实现重试，逻辑重复且质量不稳定。

### 这个项目怎么解决

支付服务把 `payment.succeeded` 作为事件写入可靠投递服务。

投递服务负责：

- 记录事件唯一键 `event_key`，避免重复创建。
- 记录每次投递尝试到 `delivery_attempts`。
- 根据 HTTP 状态码判断是否重试。
- 达到最大次数后进入 `dead`，等待人工检查或 replay。

### 能体现的能力

- 能识别“支付成功通知”这种高价值业务事件。
- 能设计幂等键，避免重复接收导致重复投递任务。
- 能把失败记录沉淀成可查询数据，而不是只依赖日志。

## 案例三：worker 重启后任务不能卡死

### 真实问题

worker 从数据库领取任务后，会把任务标记为 `running`。

如果进程刚好在发送 HTTP 请求时崩溃，这个任务可能永远停留在 `running`，后续不会再投递。

这是很多简单任务系统容易忽略的问题。

### 这个项目怎么解决

投递任务增加 `locked_until`。

worker 领取任务时设置租约时间：

```text
running + locked_until
```

如果进程崩溃，超过 `locked_until` 后，其他 worker 可以重新领取这个任务。

### 能体现的能力

- 能考虑进程异常退出，不只考虑正常流程。
- 能用租约机制解决任务卡死问题。
- 能解释为什么 `running` 状态必须有过期恢复机制。

## 面试讲法

可以按这个顺序讲：

1. 我先说明业务问题：Webhook 不是简单发 HTTP，请求失败、服务重启、下游异常都会导致通知不可控。
2. 我把一次通知拆成 event 和 delivery，两者在同一个事务里创建，保证接收事件后一定有对应投递任务。
3. 后台 worker 通过状态机处理投递任务，支持重试、死亡状态和 replay。
4. 我特别处理了 worker 崩溃导致 `running` 任务卡死的问题，用 `locked_until` 做租约恢复。
5. 最后用 delivery attempts 和 Prometheus metrics 让失败可查询、可观测。

一句话总结：

> 这个项目不是为了做一个大平台，而是聚焦解决“业务事件通过 Webhook 可靠通知外部系统”这个具体问题。
